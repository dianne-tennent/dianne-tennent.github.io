Problem Solving Techniques

25-Feb-2021

No problem too big! Here is a list of techniques to try when you just can't quite figure it out:

Pseudocode

Always helps! So many times I've tried to work out the code in my head and I've missed a key element of the problem. As soon as I start writing out what needs to be done, I can break down the problem into it's smaller parts and work it out from there. Quite often the code I write will be quite different to the pseudocode, but writing the pseudocode is still a really important step in the process.

Trying something

Definitely! No harm in hitting that run button just to see if your solution margically works. Though it's important to make sure you actually understand why it works!

Rubber ducky method

Love this method. This works really well in real life too. How many times have you started explaining your problem to someone and half way through telling them about it, you've already figured it out? Invaluable.

Reading error messages

These are becoming clearer slowly. I'm starting to understand how some methods might give me the wrong type of data, so it's important to check the documentation and make sure you're using the right method.

Console.logging

Fabulous. Always helps me to check that my code is actually doing what I think it's doing!

Googling

What a great resource! There are so many articles and tutorials out there. My favourite site is still w3schools. And Stack Overflow can be really helpful too!

Asking your peers for help

I think this is a realy good habit to get into and I try to help others as much as I can.

Asking coaches for help

Always reliable! I like the way the facilitators help you work through your own problem, rather than just give you the answer.

Improving your process with reflection

It was really helpful to revisit some of the more challenging exercises and remember where I successfully used some of the above techniques!

Putting it into practice
The problem:

I could see from the tests that the function had to handle both lowercase and uppercase versions of the word “buzz”.

The technique:

I looked at the tests to see what input was being tested so that I knew what to include.

The feeling:

I felt really proud of the way I was able to include the .toLowerCase() method within the condition so that it was catching all instances of the word “buzz”, rather than making it a separate line of code to convert the input and then check the condition separately.

The takeway:

I learned that looking at the tests helps you to write good code.

The problem:

My loop wasn’t removing the words from the list that I wanted it to.

The technique:

I googled ways of removing items from arrays and tried a few different methods. I console logged at various points in the code to find out what it was actually doing. In the end I was able to solve my problem by reading through a really great article that went through various methods for removing items from arrays. The problem was that I had skimmed the article and missed a key bit of info. When I reread it thoroughly, I found that there was a problem with using a loop to remove items from a list, because when you remove an item, then that index number disappears, and so when you increment the index with the loop, you actually end up skipping one of the items in the list. So I was able to solve that by decrementing the index if I was removing an item.

The feeling:

When I showed my code to my facilitator he said it all looked good and couldn’t figure out the problem. This made me feel good because someone who was more experienced than me was having the same trouble. Then I felt really proud when I managed to fix it by myself. Although he was really helpful and instrumental in helping me to think through the problem and isolate the area that wasn’t working.

The takeway:

I think that loops have their limitations, and that at times it may be better to use the in-built array methods such as map() and forEach(). I also learned that it’s important to know what output and parameters the inbuilt methods have.
